# Training Record - 要件定義書

## 1. プロジェクト概要

| 項目 | 内容 |
|------|------|
| プロジェクト名 | Training Record（トレーニングレコード） |
| 目的 | 個人の筋トレ・トレーニング記録を管理し、継続的な成長をサポートするWebアプリケーション |
| 対象ユーザー | 筋トレを習慣化したい個人ユーザー |
| 技術スタック | Laravel 11 / PHP 8.2 / MySQL 8.0 / Nginx / Docker / Tailwind CSS / Vite |

---

## 2. ユーザーストーリー

- ユーザーとして、アカウントを作成してログインし、自分専用の記録を管理したい
- ユーザーとして、毎回のトレーニング（種目・セット数・レップ数・重量）を記録したい
- ユーザーとして、過去のトレーニング履歴を日付やカテゴリで検索・閲覧したい
- ユーザーとして、種目ごとの重量・回数の推移をグラフで確認し、成長を実感したい
- ユーザーとして、よく使う種目をお気に入り登録して素早く記録を追加したい
- ユーザーとして、身体の部位（胸・背中・脚など）ごとにトレーニング量を把握したい

---

## 3. 機能要件

### 3.1 認証・ユーザー管理

| No | 機能 | 詳細 |
|----|------|------|
| AUTH-01 | ユーザー登録 | メールアドレス・パスワードで新規アカウント作成 |
| AUTH-02 | ログイン / ログアウト | セッションベースの認証 |
| AUTH-03 | パスワードリセット | メール経由でのパスワード再設定 |
| AUTH-04 | プロフィール編集 | ユーザー名・身体情報（身長・体重）の更新 |

---

### 3.2 種目マスタ管理（Exercise Master）

| No | 機能 | 詳細 |
|----|------|------|
| EX-01 | 種目一覧表示 | システム標準種目 ＋ ユーザー追加種目を一覧で表示 |
| EX-02 | 種目追加 | 種目名・部位カテゴリ・種別（重量/自重/有酸素）を登録 |
| EX-03 | 種目編集 | ユーザーが作成した種目の更新 |
| EX-04 | 種目削除 | ユーザーが作成した種目の論理削除 |
| EX-05 | お気に入り登録 | よく使う種目をお気に入りに追加・解除 |

**部位カテゴリ（Body Part）**

| カテゴリ名 | 主な種目例 |
|-----------|-----------|
| 胸（Chest） | ベンチプレス、ダンベルフライ |
| 背中（Back） | デッドリフト、ラットプルダウン |
| 肩（Shoulder） | ショルダープレス、サイドレイズ |
| 腕（Arms） | バイセプスカール、トライセプスプレスダウン |
| 脚（Legs） | スクワット、レッグプレス |
| 腹（Core） | プランク、クランチ |
| 有酸素（Cardio） | ランニング、ローイング |

---

### 3.3 ワークアウト記録（Workout Logging）

| No | 機能 | 詳細 |
|----|------|------|
| WK-01 | ワークアウト開始 | 日付・メモを入力してワークアウトセッションを作成 |
| WK-02 | 種目追加 | ワークアウトに種目を複数追加 |
| WK-03 | セット記録 | 各種目に対してセット数・重量（kg）・レップ数を記録 |
| WK-04 | セット追加・削除 | セットの行を動的に追加・削除（JavaScript） |
| WK-05 | ワークアウト完了 | 記録を保存して完了状態にする |
| WK-06 | ワークアウト編集 | 保存済みワークアウトの修正 |
| WK-07 | ワークアウト削除 | 記録の論理削除 |
| WK-08 | 前回記録の参照 | 同じ種目の前回セット数・重量を自動表示してガイドとして活用 |

**記録項目（種別による差異）**

| 種別 | 記録項目 |
|------|---------|
| 重量トレーニング | 重量(kg) × レップ数 × セット数 |
| 自重トレーニング | レップ数 × セット数（重量任意） |
| 有酸素 | 時間(分) または 距離(km) |

---

### 3.4 トレーニング履歴（History）

| No | 機能 | 詳細 |
|----|------|------|
| HI-01 | 履歴一覧 | 日付降順でワークアウトの一覧を表示（ページネーション付き） |
| HI-02 | 日付フィルタ | 日付範囲で絞り込み |
| HI-03 | 部位フィルタ | 特定の部位カテゴリで絞り込み |
| HI-04 | 詳細表示 | 1回分のワークアウトの全種目・全セットを表示 |

---

### 3.5 統計・進捗管理（Statistics）

| No | 機能 | 詳細 |
|----|------|------|
| ST-01 | 種目別ボリューム推移 | 特定種目の1RM推定値・合計ボリューム（kg×rep）の時系列グラフ |
| ST-02 | 部位別トレーニング頻度 | 月次・週次での部位ごとのトレーニング回数の可視化 |
| ST-03 | 継続ストリーク | 連続トレーニング日数・今月のトレーニング回数の表示 |
| ST-04 | 体重推移（任意） | 体重の時系列グラフ |

---

### 3.6 ダッシュボード（Dashboard）

| No | 機能 | 詳細 |
|----|------|------|
| DB-01 | 直近のワークアウト | 最新3〜5件のワークアウト一覧 |
| DB-02 | 今週のサマリー | 今週のトレーニング回数・合計ボリューム |
| DB-03 | 連続記録（ストリーク） | 継続日数の表示 |
| DB-04 | クイック記録ボタン | ワークアウト新規作成への導線 |

## 4. 非機能要件

### 4.1 パフォーマンス
- ページレスポンスタイム：通常操作で 1秒以内
- グラフ描画：2秒以内

### 4.2 セキュリティ
- CSRF トークンによるフォーム保護
- 認証済みユーザーのみ記録を操作可能
- 他ユーザーのデータへのアクセス禁止（ポリシーによる制御）
- パスワードは bcrypt でハッシュ化

### 4.3 ユーザビリティ
- レスポンシブデザイン（スマートフォン・タブレット・PC 対応）
- スマートフォンでの片手操作を考慮したUI設計
- 直感的な操作フロー（3タップ以内でセット記録を完了）

### 4.4 信頼性
- データの論理削除による安全な削除管理
- バリデーションによる不正データの排除

## 5. データベース設計（概要）

### 主要テーブル

```
users                   ユーザー情報
  └─ id, name, email, password, height, weight, ...

body_parts              部位カテゴリマスタ
  └─ id, name, slug

exercises               種目マスタ
  └─ id, user_id(nullable), body_part_id, name, type(weight/bodyweight/cardio), is_default

user_favorite_exercises ユーザーのお気に入り種目
  └─ user_id, exercise_id

workouts                ワークアウトセッション
  └─ id, user_id, date, memo, status(in_progress/completed)

workout_exercises       ワークアウト内の種目
  └─ id, workout_id, exercise_id, order

sets                    セット記録
  └─ id, workout_exercise_id, set_number, weight, reps, duration_min, distance_km

body_weight_logs        体重記録（任意機能）
  └─ id, user_id, date, weight
```

## 7. 開発フェーズ

### Phase 1（MVP）
- ユーザー認証（登録・ログイン・ログアウト）
- 種目マスタ（デフォルト種目のシード）
- ワークアウト記録（作成・一覧・詳細・削除）
- セット記録（重量・レップ数の入力）

### Phase 2
- ダッシュボード
- 履歴フィルタ・検索
- ユーザー独自の種目追加
- お気に入り種目
- 前回記録の参照

### Phase 3
- 統計グラフ（種目別ボリューム推移）
- 部位別トレーニング頻度
- 継続ストリーク
- 体重記録

## 9. 制約・前提条件

- 本アプリは個人利用を想定した**シングルテナント**設計（マルチテナント対応は対象外）
- 初期リリースはWeb版のみ（モバイルアプリは対象外）
- 外部APIとの連携は対象外（将来的な拡張として検討可能）
- Docker環境でのローカル開発を前提とする（本番デプロイは別途検討）
